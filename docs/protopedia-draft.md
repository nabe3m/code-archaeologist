## 作品名

**Code Archaeologist — コードの歴史を発掘し、「消せないコード」に引導を渡す AI エージェント**

## 概要

git blame から PR・Issue まで自律的に遡り、コードの「なぜ」を出典付きで解明。役目を終えた防御的コードには証拠付きの削除 PR と再発予言を添える AI エージェント。

## ① 課題と背景

**「このコード、消していいですか?」に、誰も答えられない。**

歴史のあるコードベースには、目的の分からない sleep、理由不明のリトライ、誰も触れない条件分岐が積み重なっています。書いた本人はすでに退職し、意思決定の経緯は数年前の PR 議論や Issue のやり取りの奥に埋もれている。`git blame` をたどっても、返ってくるのはコミットメッセージ1行だけです。

その結果、次のことが起こります。

- **消せない**: 理由の分からないコードを消す勇気は、誰にも持てません。「触らぬ神に祟りなし」で技術的負債が固定化します
- **遅くなる**: 役目を終えた防御的コード（例: 外部 API の不具合を避けるための待機処理）が、その API が直った後も全ユーザーの時間を奪い続けます
- **暗黙知が失われる**: 設計判断の「なぜ」が、オンボーディングのたびに口伝で再構築されます

属人化・退職・異動が避けられない以上、これはすべての開発組織に共通する DevOps の根源的な課題です。そして、書いた本人と対話できず、判断の履歴がすべて PR / Issue として公開されている OSS こそ、この「歴史の発掘」が最も効く場でもあります。

なお、私自身は特定のパブリック・ブロックチェーンを広める活動をしながら、その周辺で少しずつ OSS にも関わっています。そこで実装の履歴を追う作業が毎回とても大変で、「なぜこのコードはこうなっているのか」を人力で掘り続けることに限界を感じたことが、このアイデアの出発点になりました。

## ② 想定ユーザー

- **歴史のあるコードベースを運用する開発チーム**: レガシー刷新やリファクタリングの判断に、確かな「証拠」が欲しい
- **新規参画メンバー**: 「なぜこうなっているのか」を、先輩の記憶に頼らず自分で調べたい
- **テックリード / EM**: 技術的負債の棚卸しで、「消してよいコード」を安全に見極めたい
- **OSS コントリビューター / メンテナ**: 初見のリポジトリでも、その実装が生まれた経緯を自分で掘り当て、根拠を引用しながら安心して PR を送りたい

## ③ 特徴・工夫

**1. 自律遡行 — エージェントである必然性**

対象のコード行から git blame → コミット → PR → 議論 → Issue へと遡っていく際、**「次にどこを掘るか」を毎ステップ Gemini が自律的に判断**します（あらかじめ決めた固定パイプラインではありません）。掘り進める先はコードの歴史の形によって毎回変わるため、探索の主導権をエージェント自身が握ることが本質的に重要になります。判断の全過程は構造化イベントとして UI にリアルタイム配信され、「AI がいま何を考えて掘っているか」がそのまま見えます。

**2. 出典付き回答 — ハルシネーションと戦う設計**

史官エージェントの回答には、すべての主張に実在の PR / Issue / コミットへの引用 [n] が付きます。証拠が見つからなければ「分からない」と正直に答え（証拠がゼロのときは LLM に回答させない）、失敗したツール呼び出しの結果は LLM にそのままフィードバックし、早すぎる調査終了は一度差し戻す——ライブ検証で見つけた LLM の失敗パターンを、プロンプトの言い回しではなく**仕組み**で潰しています。

**3. 失効コードの削除 PR 自動作成＋🔮予言 — 発掘から行動、そして安全網まで**

監査官エージェントは「存在理由がすでに失効した防御的コード」を見つけると、**発掘した証拠を引用した削除 PR を GitHub 上に実際に作成**します。「v2 に移行したら消す」という3年前の約束を、人間の代わりにエージェントが果たすわけです。さらに🔮予言者 Oracle が、そのコードが**かつて防いでいた障害**と、再発したときの兆候・対処を、削除 PR への注意コメントとして書き残します——「消してよい、ただし歴史を知った上で」。実例: https://github.com/nabe3m/demo-repo/pull/14 （[Oracle の予言コメント](https://github.com/nabe3m/demo-repo/pull/14#issuecomment-4936010078)）

引用と根拠を備えたこの削除 PR は、社内リポジトリの負債返済にとどまらず、**そのまま OSS への貢献**にもなります。メンテナが最も知りたい「なぜこのコードは要らなくなったのか」に、公開された一次資料へのリンクで答える PR は、レビューの土台がすでに整った状態で届きます。

**4. AI の品質をデプロイ前に守る evals ゲート — DevOps × AI の要**

回答品質を「正しい一次資料を引用できたか」で採点する独自の evals（考古学クイズ5問。LLM の文言のゆらぎに左右されない採点方式）を、**Cloud Build のデプロイ前ゲート**として実行します（4/5 未満ならデプロイを中止）。これにより、プロンプトやモデルの変更で生じたデグレが本番に届きません。このゲートは、開発中に実際のバグを**3つ**捕まえました。最後の1つは「エージェント自身が作った削除 PR が検索結果を汚染し、史官が一次資料ではなく自分の過去の要約を引用しはじめる」という自己汚染で、エージェントが自律的に成果物を増やすからこそ起きる挙動です。evals の失敗として現れなければ、まず気づけませんでした。

**5. 実装**

Google Cloud Run（単一サービス・SSE ストリーミング）+ Gemini（判断は 2.5 Flash、回答は 2.5 Pro と使い分け）。**LLM の呼び出しは Vertex AI 経由（Workload Identity で鍵レス）**で、環境変数の切り替え1つで Developer API へフォールバックもできます。遡行ループのコアは TDD で作り込み（48テスト）、push → evals ゲート → Cloud Run という CD を構築しました。取得結果は2層キャッシュでレート制限を回避し、シークレットは Secret Manager で管理しています。

## システム構成

（docs/architecture.png を貼付）

全体は **Cloud Run 上の単一サービス**にまとまっています。フロントの React UI から発掘を開始すると、リクエストは1つのバックエンドに入り、そこで**調査官 → 史官 → 監査官 → 🔮Oracle** の各エージェントが順に働きます。この間の判断や掘り先の遷移は **SSE（Server-Sent Events）でブラウザへ逐次ストリーミング**され、UI のタイムラインが「いま何を掘っているか」をリアルタイムに描き続けます。サービスを割らずに1つに収めたことで、状態の受け渡しやデプロイがシンプルになり、SSE の長時間接続もそのまま扱えます。

**LLM は Vertex AI 経由で呼び出します。** 認証は Workload Identity による鍵レス方式で、サービスアカウントキーをコードにもコンテナにも持ちません。モデルは用途で使い分け、遡行の「次にどこを掘るか」という判断には応答の速い **Gemini 2.5 Flash**、証拠を束ねて出典付きで語る最終回答には推論力の高い **Gemini 2.5 Pro** を割り当てています。環境変数の切り替え1つで Vertex AI と Developer API を行き来でき、ローカル開発と本番で同じコードが動きます。

**GitHub API へのアクセスは、必ず2層キャッシュ（メモリ + ディスク）を通します。** blame・コミット・PR・Issue の取得結果をキャッシュし、同じ発掘の中で何度も同じ資料を引くレート制限を回避します。加えて、公開エンドポイントを匿名で高コストな調査に濫用されないよう、**同時実行スロットと IP あたりのレート制限（プロセス内 Throttle）** を入れ、Cloud Run の `--max-instances` と併用して総コストに上限を設けています。シークレットはすべて Secret Manager で管理し、コンテナには実行時に注入されます。

デプロイは **push → evals ゲート（Cloud Build）→ Cloud Run** の一本道です。回答品質のリグレッションを検知する evals をデプロイ前ゲートに置くことで、AI の振る舞いのデグレが本番に到達する前に止まります。
